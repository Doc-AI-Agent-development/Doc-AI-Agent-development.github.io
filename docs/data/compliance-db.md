# 컴플라이언스 DB (관계형)

### 정의

각 팀이 실시해야 하는 교육의 법령·절차상 의무 기준을 **정확 조회 가능한** 형태로 담은 표입니다. 두 시나리오(연간계획·교육자료)의 출발점입니다.

### 담는 정보 (개념)

- **실시 의무**: 어떤 팀이 어떤 교육을 어떤 주기·시간으로 실시해야 하는지 — 팀과 교육을 잇는 정보.
- **교육 기준(필수항목)**: 그 교육이 다뤄야 할 필수항목과 근거 — 교육 자체에 붙는 정보로, 어느 팀이 실시하든 동일.

### 실제 구조 (NDATA 평탄 스키마)

외부 DB(NDATA)가 제공할 형태에 맞춰 **평탄(denormalized)** 하게 둡니다. 별도 "교육 기준" 테이블 없이, 교육 메타를 실시 의무 행에 inline으로 담습니다.

| 테이블 | 내용 |
| :-- | :-- |
| `team_obligations` | **팀 × 교육** 한 행. 교육 메타(과정명·구분·주기·법정시간·대상자·근거)가 **inline**. PK = (team_id, edu_id) |
| `education_checklist` | 교육별 **필수항목**(체크리스트). edu_id로 묶임 |
| `users` | 사용자 → 소속 팀 (BFF가 user_id 전달) |
| `documents` · `education_documents` · `document_contents` | 교육자료 저장소(라우터·자료 레이어) — [교육자료 저장소](./content-repository.md) 참고 |
| `annual_plans` | 작성된 연간계획 보관 (BFF 미구현 임시 대역) |

### 활용

- **연간계획**: 담당자의 팀을 정하면, 그 팀의 `team_obligations`(실시해야 할 교육 목록·주기·시간·근거)를 조회합니다. 이것이 연간계획표의 뼈대.
- **교육자료**: 그 목록에서 만들 교육 하나를 고르면, 그 교육의 메타(`team_obligations`)와 필수항목(`education_checklist`)을 조회합니다. 필수항목이 목차의 뼈대이자, 다 만든 뒤 빠진 주제 점검 기준.

### 설계 근거

- **왜 평탄한가**:
  1. 외부 NDATA DB가 줄 형태와 일치 → 실 DB로 교체할 때 조회 도구 내부만 바꾸면 됨.
  2. 에이전트(LLM)는 SQL·JOIN을 직접 하지 않고 **도구가 조립한 결과**만 받음 → 평탄이어도 LLM엔 영향 없음(오히려 한 행으로 완결, JOIN 실수 여지 없음).
  3. 데이터 규모가 작아 중복(교육 메타가 팀마다 반복) 비용이 미미. 단 **1:다 관계(필수항목)**는 행 폭증을 피해 `education_checklist`로 분리.
- 법령 기준·사내 규칙은 "비슷한 것"이 아니라 정확히 그 기준이어야 하므로, 의미검색(벡터)이 아닌 **정확 조회**로 둡니다.
- 교육-문서 매핑은 여기 두지 않고 [교육자료 저장소](./content-repository.md)의 라우터 레이어에 위임합니다 — 자료가 추가·교체돼도 기준이 흔들리지 않도록.

### 접근

- 에이전트의 자율 탐색·자유 SQL을 막고, 사전 정의된 **조회 도구**로만 접근합니다. → [SQL 조회 툴](../tools/sql-query-tool.md)
